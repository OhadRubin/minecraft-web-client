# Visual Sketchpad and Visual Chain-of-Thought in LLM Reasoning

## Visual Sketchpad: Enabling LLMs to Draw as a Chain-of-Thought

The *Visual Sketchpad* framework (Hu *et al.* 2024) introduces a paradigm where a multimodal large language model can generate **intermediate visual sketches** (drawings, plots, markups) as part of its reasoning process. Instead of relying only on text-based chain-of-thought, the LLM is given a “sketchpad” tool to draw lines, shapes, boxes, and marks on a canvas, similar to how humans draw auxiliary lines or diagrams to solve problems. For example, when asked to prove that the angles of a triangle sum to 180°, the model can draw the triangle and add an auxiliary line to help in the geometric proof. These visual doodles become part of the model’s reasoning: the LLM plans its solution by both generating text and examining the drawn visual artifact it created.

Critically, Sketchpad’s drawing actions are implemented via code generation – the LLM produces Python code (using libraries like Matplotlib or OpenCV) to render the sketches. This allows it to plot functions on a graph, draw geometric figures, or highlight image regions as needed during reasoning. Such **visual chain-of-thought** can handle a range of tasks: the authors demonstrated improvements on math problems (geometry, function plotting, graph theory, even chess boards) and on vision-language tasks that benefit from marking up images. In experiments, adding a sketching ability yielded substantial performance gains: *Visual Sketchpad* improved math problem-solving accuracy by \~12.7% and vision reasoning by \~8.6% on average compared to text-only reasoning. In fact, an LLM (GPT-4o) equipped with Sketchpad achieved new state-of-the-art results on benchmarks like V\*Bench visual reasoning and BLINK spatial reasoning, outperforming even stronger baseline models without sketching. These results indicate that giving LLMs a way to “draw” their thoughts can significantly boost reasoning accuracy.

Notably, the Sketchpad framework also allows the LLM to incorporate **specialist vision models** as tools during its visual reasoning. For instance, the LLM can call an object detection model (like Grounding DINO) to automatically draw bounding boxes around objects in an image, or use a segmentation model (like Segment Anything) to sketch a mask over a region. By integrating these pre-trained vision modules into its sketching process, the LLM gains access to reliable perceptual operations – it can mark and label parts of an image which it might not accurately detect from raw pixels alone. This combination of drawn visuals and specialist models further improves the model’s visual reasoning and perception. In summary, the original Visual Sketchpad work demonstrated that an LLM with a “visual scratch pad” for intermediate drawings can plan solutions more effectively, much like a person using pen and paper, yielding both **better accuracy and a more interpretable reasoning process**.

## Extending Visual Sketching to 3D Spatial Reasoning

A natural extension of the visual sketchpad idea is to apply it to **3D spatial reasoning tasks**, where problems involve three-dimensional relationships or navigation in space. Recent research has indeed begun to explore methods for eliciting spatial understanding in LLMs by generating *visual or spatial representations* as intermediate steps. For example, Wu *et al.* (2024) propose the *Visualization-of-Thought (VoT)* technique – nicknamed the *“Mind’s Eye”* of LLMs – which prompts the model to produce imagined visualizations to guide multi-step spatial reasoning. In tasks like navigation instructions or block arrangement in a grid, the LLM interleaves its text thinking with descriptions of a mental image or map, effectively drawing out a scene in its “mind’s eye.” This approach significantly improved performance on multi-hop spatial tasks (like 2D grid navigation and visual tiling puzzles), **outperforming even state-of-the-art multimodal LLMs** on those benchmarks. By visualizing its reasoning trace (e.g. sketching a path on a grid world or imagining the layout of a room), the model was better able to handle spatial relations and multi-step spatial logic. The VoT results suggest that *even without an actual rendered image*, prompting an LLM to articulate a spatial diagram or mental picture can enhance its 3D reasoning capabilities, analogous to how humans imagine scenes to solve spatial problems.

Going a step further, other work has looked at giving LLMs tools to handle real 3D data and visualizations. One direction is to enable an LLM to generate **3D plots or scene renderings** via code, extending the sketchpad concept into three dimensions. While dedicated “3D sketchpad” systems are still nascent, one can envision an LLM using libraries like Three.js or Matplotlib 3D to draw a 3D coordinate axis, plot shapes, or simulate spatial configurations as it reasons. For instance, an LLM solving a geometry word problem about a polyhedron could write Python code to render that 3D shape and mark angles or distances, then use the rendered model to verify relationships. Some experimental works hint at this capability – e.g. researchers have fine-tuned LLMs to generate Three.js scripts for scene construction, treating code as a means to **visualize understanding in 3D space** (though these are often for final outputs rather than chain-of-thought). More formally, to test 3D spatial understanding, Daxberger *et al.* (2025) introduced an MLLM called *MM-Spatial* that was trained on indoor 3D scene data and depth information. While MM-Spatial does not have the model literally sketch in 3D, it supports a form of spatial chain-of-thought reasoning by leveraging depth maps and multi-view images as additional inputs. The model can call a depth estimation tool to get metric distances, reason about “in front of / behind” relations, and even predict 3D object bounding boxes, using the depth input as a kind of spatial *visual aid* during reasoning. This indicates that bridging LLMs with 3D perception tools (like depth sensors or multi-view imagery) can imbue them with better 3D awareness.

In summary, the visual chain-of-thought paradigm is being extended beyond 2D diagrams to the realm of **3D spatial reasoning**. Approaches like VoT show that prompting LLMs to conjure internal visualizations boosts spatial cognition, and systems like MM-Spatial demonstrate that LLMs can use 3D data (depth, multi-view scenes) as part of a tool-assisted reasoning chain. We are likely moving toward LLMs that not only sketch on a 2D canvas but also manipulate and “imagine” 3D environments – for example, future models may generate intermediate 3D models or point-clouds to solve complex spatial puzzles, bringing the **visual sketchpad** into three dimensions.

## LLMs Generating Visualizations via Code for Reasoning

Another thread of research focuses on **LLMs generating code to produce visual outputs** (charts, graphs, diagrams) as a way to work through problems. This is related to program-aided reasoning, where an LLM writes and executes code to help solve a task. In the context of visual reasoning, the code written by the model produces a visualization that the model can then use to infer an answer. The Visual Sketchpad system itself is a prime example: the model writes code to draw geometry figures or plot function graphs, effectively calling libraries like Matplotlib or OpenCV as “visualization tools” in its chain-of-thought. For instance, to solve a calculus question about a function’s behavior, an LLM could generate a quick Matplotlib plot of the function and inspect it to determine intervals of increase/decrease. Likewise, for a graph theory puzzle, the LLM might use the NetworkX library to programmatically draw a graph and label nodes, helping it visualize the structure before reasoning about paths or cycles. These intermediate visualizations are not the final answer presented to a user, but rather **internal aids** that the LLM uses to get to the answer (analogous to scratch work on a notepad).

There have been several efforts in this vein. Chen *et al.* (2023) introduced *Program-of-Thought* prompting, where the model intermixes natural language reasoning with code generation, executing the code to verify steps or compute results. In many cases this code can produce visual output. For example, *CodeGraph* (Hao *et al.* 2024) is a method where an LLM solves graph reasoning problems by generating Python code that constructs and analyzes graphs. Instead of relying purely on textual logic (which can be error-prone for complex graphs), the LLM writes a short program to, say, build the graph and calculate properties (connected components, shortest paths) or even display it. This led to notable accuracy improvements on graph-theoretic QA tasks, demonstrating the benefit of offloading work to code execution. Similarly, for mathematical reasoning, previous studies have shown that having the LLM draw a plot or diagram via code can catch mistakes. If the model is unsure about a calculus question, plotting the function might reveal a pattern that a text-only chain-of-thought could miss. Essentially, by **“showing its work” in code and visuals**, the LLM can achieve more reliable reasoning.

It’s important that these visuals are used as *interactive tools* in the reasoning loop, not just as pretty outputs. In the Visual Sketchpad framework, after the LLM draws something, it actually consults that drawing for the next step of reasoning. This might be implemented by feeding the generated image (or its description) back into the multimodal model. For example, the LLM draws an auxiliary line on a triangle, then “sees” the updated diagram and deduces the next geometric relation from it. This interactive, iterative use of visual tools is akin to how a person might draw, look, then draw more until the problem is solved. We are seeing the emergence of **LLM agents that dynamically create visual artifacts (plots, graphs, diagrams) through code as part of a reasoning pipeline**, rather than relying solely on internal symbolic manipulation. This approach leverages the precision of code execution (for calculations and drawing) to complement the LLM’s language-based reasoning, thus reducing errors (especially in domains like math) and providing a form of visual checking or intuition.

## Tool-Using LLMs with Vision Models and Visual Annotations

In addition to generating their own drawings, LLMs can also **call external vision tools** to analyze or annotate existing images as a step in reasoning. Pioneering systems like *Visual ChatGPT* and *HuggingGPT* demonstrated this capability: a language model orchestrates a suite of image-processing models (for example, object detectors, segmenters, image captioners) to tackle a complex task. During a multi-step query, the LLM can decide to invoke a tool like “segment this image” or “detect objects here,” use the output (e.g. a mask or bounding boxes), and then continue the reasoning with that new visual information. This effectively gives the LLM a way to *augment its chain-of-thought with visual perceptual acts*. For instance, if asked “How many people in this photo are wearing glasses?”, the LLM might call a face detection model and a glasses classifier on each face, mark the detections on the image, count the ones with glasses, and return the answer. The intermediate stage of drawing boxes around faces with glasses is a form of visual reasoning step, making the process more transparent and grounded.

The Visual Sketchpad framework also embodies this idea by integrating **specialist vision models into the drawing process**. As noted, Sketchpad can use an object detection model to draw a bounding box on the sketchpad, or a segmentation model to fill in a region. Those annotated sketches then inform the LLM’s next steps. This kind of tool usage is especially useful for visual tasks that require pinpointing specific image regions (e.g. “Find the red circle in the image and describe what’s inside it”). Instead of relying purely on its internal image features, the LLM delegates the localization to a powerful vision model, gets an explicit annotation (like a box coordinates or mask), and reasons over that. Recent research formalizes this approach: Fan *et al.* (2025) introduced **GRIT**, a method for teaching multimodal LLMs to interleave text and *explicit bounding box coordinates* in their reasoning chains. In GRIT, the model’s chain-of-thought might say, *“<think> The relevant object is \[bbox: (50,80,120,150)]. It’s a person holding a balloon.”* – thereby referencing a specific region of the image as part of its thought process. These coordinates act as pointers to visual evidence, making the reasoning both grounded and interpretable. Impressively, GRIT trains models to produce such *mixed text-visual* chains using reinforcement learning, without needing thousands of annotated thought traces. The result is an LLM that can **“think with images,”** not just about images, by literally marking up regions it considers important during inference.

Beyond bounding boxes, similar approaches use other visual annotations in reasoning. Some works use segmentation masks or keypoints as intermediate outputs to solve tasks like reasoning segmentation (where a complex instruction needs objects segmented step-by-step). Depth estimation models can also be tools for an LLM: for example, in a 3D reasoning scenario the LLM could call a depth predictor on an image and use the resulting depth map to reason about distances (“the car is closer than the house based on depth data”). In fact, the MM-Spatial model from Apple is designed to incorporate **metric depth inputs via tool-use** to answer questions about object sizes and distances in an image. By plugging in a depth estimation module, the LLM gains a quantitative visual sense that pure vision-language models lack. We see a common theme here: *tool-using LLMs* can dynamically invoke vision algorithms (detection, segmentation, depth, OCR, etc.) as part of their reasoning chain to overcome limitations of their end-to-end learned perception. This results in a more powerful and modular system. The LLM becomes a coordinator that reasons about **which visual tool to use when**, and integrates the tool’s output (often by drawing or annotating) into its chain-of-thought. Such systems have achieved stronger results on tasks like complex visual question answering, because they combine the broad reasoning ability of LLMs with the precision of dedicated vision models.

## Benefits of Visual Chain-of-Thought vs. Text-Only Reasoning

Integrating a “visual sketchpad” into LLM reasoning offers multiple benefits over the conventional text-only chain-of-thought. First, it helps **offload cognitive burden**. Just as humans sketch to extend their limited working memory, an LLM can use an external drawing to keep track of complex information or relationships that would be hard to juggle in text alone. Diagrams can make certain implicit spatial or mathematical relationships explicit. By externalizing part of the thought process into a picture, the model doesn’t have to infer everything from textual descriptions – the visual serves as an extra memory. The Visual Sketchpad paper explicitly notes that sketching “amplifies our ideas and relieves our limited-capacity working memory,” which is missing from today’s models that reason only with text. Thus, visual chain-of-thought can reduce mistakes that come from losing track of details in a long textual reasoning sequence.

Secondly, visual intermediate steps can **clarify ambiguity and improve accuracy**. Many reasoning mistakes by LLMs (especially in spatial or mathematical domains) come from mis-visualizing the scenario described in text. When the model actually draws the scenario (or has access to a diagram), it constrains the reasoning to something consistent with the drawing. For example, an LLM might misestimate an angle relationship if it only has text, but if it draws the geometry, it can measure or visually confirm the angle sum property. Empirically, we saw that models like Sketchpad and VoT yield higher solution rates on complex tasks by leveraging visuals. VoT’s “mental images” greatly enhanced spatial navigation reasoning, indicating that even a rough visualization prevents certain logical errors. Likewise, in graph problems or arithmetic, having the model generate a chart or run a small program can catch arithmetic errors or logical leaps, leading to more **robust and verified reasoning**.

Another benefit is **improved interpretability and grounding** of the model’s thought process. Visual steps (like a drawn bounding box or a plotted curve) provide a transparent glimpse into what the model is attending to. This addresses the “black box” concern where a text-only chain-of-thought might assert something without evidence. With visual chain-of-thought, one can literally see the evidence the model used – e.g. *“it highlighted this region of the image and based its answer on that”*. Fan *et al.* point out that purely text-based reasoning for vision tasks lacks explicit visual integration, making it hard to trust if the model truly looked at the right part of the image. By interleaving bounding boxes or other visual tokens in the reasoning, the model’s decision path becomes clearer and *visually grounded*. This can increase user trust and make debugging the model’s logic easier (we can identify if the model drew the wrong conclusion perhaps by pointing at the wrong thing in an image). Researchers have begun to create benchmarks that specifically require this kind of interpretable visual chain-of-thought – for instance, a recent dataset provides questions where the model must output an intermediate bounding box pointing to the evidence before answering.

Finally, visual reasoning steps bring **human-like problem-solving strategies** into AI. Humans have long used sketches, diagrams, and written calculations on paper as tools to think through problems; those externalizations are part of our intelligence. Enabling LLMs to do something analogous (be it drawing a mini-diagram or calling a visualization tool) moves them closer to human cognitive patterns. This cross-modal thinking could unlock new abilities in LLMs, especially for tasks that are innately visual or spatial. Early evidence shows that when allowed to sketch, models become more robust – Hu *et al.* report that Sketchpad made the model’s performance not only higher but also *more consistent* and adaptable across different kinds of tasks. In sum, augmenting LLM reasoning with a **“visual chain-of-thought”** yields better performance and aligns with how humans solve complex problems, all while providing a clearer window into the model’s thought process. The convergence of language, code, and vision in these systems is a promising path toward more general and reliable AI reasoning.
